# `.agents/` — agent self-describing layer

> **What this folder is.** The repository-local layer that lets
> AI agents (and humans) navigate the project, write new rules,
> grow a memory, and contribute new skills without polluting the
> codebase with random files. Everything here is governed by
> `AGENTS.md §3` (the role-boundary table).
>
> **Nothing under this folder is "the agent's private notes".**
> Everything is a first-class tracked file that ships with the
> repo and lives by the same rules as everything else.

## Layout

```
.agents/
├── README.md                              ← you are here
├── RULES.md                               ← index of every rule that governs a .md
├── rules/                                 ← one .md.rule per tracked .md file
│   ├── agents.md.rule                     ← appliesTo: AGENTS.md
│   ├── readme.md.rule                     ← appliesTo: README.md
│   └── …
├── skills/                                ← repo-local skills
│   ├── navigating-project-docs/
│   │   └── SKILL.md                       ← procedure: read/update product docs
│   └── maintaining-agents-dir/
│       └── SKILL.md                       ← procedure: maintain rules/skills/memory
└── memory/
    ├── README.md                          ← how to add a memory entry
    ├── facts/   (.gitkeep)                ← facts we learned
    ├── retros/  (.gitkeep)                ← retrospectives on mistakes
    ├── lessons/ (.gitkeep)                ← behaviour-changing lessons
    └── index.tsv                          ← generated by `bun run memory:reindex`
```

## Roles (one line each)

| File | Role | Read first? |
|---|---|---|
| `README.md` (this file) | "How is `.agents/` organised?" | Yes, if you are new. |
| `RULES.md` | "What rule governs what file?" | Yes, before writing any `.md`. |
| `rules/*.md.rule` | "Why does this file exist and where does it sit?" | Per file, when reviewing. |
| `skills/*/SKILL.md` | "How do I do X?" | When the skill's `when to use` matches your task. |
| `memory/facts/*.md` | "What do we already know?" | Before stating a "new" fact. |
| `memory/retros/*.md` | "What did we get wrong, and what changed?" | After a bug or a rework. |
| `memory/lessons/*.md` | "What behaviour should I change from now on?" | When the lesson's `applies when` matches your context. |
| `memory/index.tsv` | grep-friendly index of all memory entries | Generated; never edit by hand. |

See `AGENTS.md §3` for the full role-boundary table and the
decision tree for "where does my new file go?".

## How to add things

- **Add a memory fact / retro / lesson** → load
  `.agents/skills/maintaining-agents-dir/SKILL.md`.
- **Add a rule for a new tracked `.md`** → load
  `.agents/skills/maintaining-agents-dir/SKILL.md`.
- **Add a skill** → load
  `.agents/skills/maintaining-agents-dir/SKILL.md`.
- **Add or update a product doc** (a doc outside `.agents/`)
  → load `.agents/skills/navigating-project-docs/SKILL.md`.

## How the check scripts work

`bun run check:md` enforces "every tracked `.md` has a rule". It
walks `git ls-files '*.md'`, parses the `appliesTo` glob from
each row in `RULES.md`, and fails on any unmatched `.md` outside
the explicit allowlist (`AGENTS.md`, `.agents/RULES.md`,
`.agents/rules/*.md.rule`, `.agents/skills/*/SKILL.md`).

`bun run check:files` enforces "every `.ts`/`.tsx`/`.json` under
`packages/` has a sibling `README.md` or is allowlisted". It
catches new source files that arrive without a per-package doc.

`bun run memory:reindex` walks `.agents/memory/**/*.md`, parses
frontmatter, and emits `.agents/memory/index.tsv`. Run it after
adding memory entries; it is also part of `bun run ci`.

## What does NOT live here

- **Product code.** Lives in `packages/<name>/src/`.
- **Product documentation.** Lives in `SPEC.md`, `ROADMAP.md`,
  `CHANGELOG.md`, or `docs/adr/` — once those exist.
- **Build tooling.** Lives in `package.json`, `tsdown.config.ts`,
  `.oxlintrc.json`, `.prettierrc.json`, `tsconfig.base.json`,
  `bunfig.toml`, `vitest.workspace.ts` at the repo root, and in
  CI workflows at `.github/workflows/`.
- **CI workflows and GitHub config.** Lives in `.github/`.

If something does not fit any of the above, ask: "is it really
needed, or am I about to create a random file?" Then check the
skills and only then decide.
